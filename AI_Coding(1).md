# AI Coding 工程约束：AI 项目上下文同步规范

> **目标**：在 AI 辅助编程（Vibe Coding）中，确保 AI 始终基于**最新、最准确的项目全貌**进行协作，消除因上下文过时导致的错误与幻觉。  
> **核心机制**：以项目根目录下的 `PROJECT_CONTEXT.md` 为唯一事实来源，**在用户要求 git push 时自动完成差异分析、上下文更新与推送**，实现代码与文档的原子性同步。  
> **辅助手段**：利用 Git 提交记录规范化记录需求迭代过程，让 AI 可独立读取需求演进历史，辅助判断上下文差异。

---

## 1. 项目说明文档（PROJECT_CONTEXT.md）

### 1.1 定位与存放
- 每个项目根目录下必须存在且仅存在一份 `PROJECT_CONTEXT.md`。
- 该文档**专门写给 AI 阅读**，兼顾人类可维护性，格式高度结构化。
- 纳入版本控制（Git），随项目代码一同演进。

### 1.2 文档结构（AI 优化版）
以 YAML Front Matter + 结构化 Markdown 组织，便于 AI 快速提取关键元信息。**所有章节均为必填，不可省略。**

```yaml
---
# 元数据（AI 优先读取）
project_name: MyProject
version: 1.2.0
last_updated: 2026-04-29T10:30:00Z
language: TypeScript
framework: Next.js 14
database: PostgreSQL 16
ai_model_preference: DeepSeek
---
```

#### 必含章节及 AI 读取指引
| 章节                           | 用途                            | AI 关键动作                   |
| ---------------------------- | ----------------------------- | ------------------------- |
| **1. 项目目标与范围**               | 一句话核心价值 + 明确边界                | 判断任务是否超出范围，防止范围蔓延         |
| **2. 技术栈与版本**                | 语言、框架、关键库及精确版本                | 生成代码时锁定 API 兼容性，避免使用已弃用特性 |
| **3. 目录结构与职责**               | 树形目录 + 各目录/关键文件职责说明           | 新增文件时正确定位，修改时先理解影响面       |
| **4. 编码规范与约定**               | 命名规则、文件组织、状态管理、API 设计等显式约定    | 生成代码时严格遵循，保持项目一致性         |
| **5. 当前功能状态**                | 各功能模块完成度（✅已完成 / 🔄进行中 / ⏳待开发） | 避免重复实现已有功能，评估改动关联性        |
| **6. 已知问题与限制**               | 技术债务、待修 Bug、性能瓶颈、第三方限制        | 新方案不引入已知问题，必要时先提议修复       |
| **7. 待办事项（Product Backlog）** | 按优先级排序的下一步工作                  | 用户提出新需求时，智能关联已有待办，建议优先级   |
| **8. 变更日志**                  | `版本 / 日期 / 变更摘要 / AI 或人` 表格   | 追踪演进路径，理解“为什么这样改”         |
| **9. AI 对话摘要（可选）**           | 最近几次关键决策与对话结论的摘要              | 新会话快速还原上下文，减少重复沟通         |

> **格式约束**：全部使用列表、表格、代码块；禁止出现模糊的长段散文描述。每一个事实性描述必须可被正则或简单 AST 解析。

## 2. 同步机制（核心闭环：Push 即同步）

### 2.1 唯一触发时机
**仅当用户明确要求 AI 执行 `git push` 时**，AI 才启动完整的上下文同步流程。其他任何时刻（包括对话中提出新需求、修改代码）AI 均**不主动打断**请求更新文档，而是将差异观察暂存于对话上下文中。

### 2.2 Push 前自动同步流程
当用户指令包含 `git push` 或语义等同操作时，AI 必须按以下顺序自动执行，完成后才真正推送：

1. **对话需求提取**：回顾本次对话全过程，提取用户提出的所有**实质性新增需求或改动意图**，归纳为结构化要点（如：“新增用户头像上传功能”“重构支付状态机”）。

2. **Git 历史分析**：执行 `git log --oneline --grep="feat|fix|refactor" -15`，获取近期功能性提交，识别可能尚未反映到文档中的已完成/进行中需求。

3. **实际项目扫描**（与 2.3 节扫描项相同）：
   - 比对 `package.json` / `Cargo.toml` / `requirements.txt` 等依赖文件中的版本与文档记录。
   - 检查文档目录结构是否与现实一致。
   - 若项目配置了测试/构建脚本，可选运行轻量级检查（如 `tsc --noEmit`）。

4. **综合差异生成**：将 **对话需求 + Git 历史未记录需求 + 实际扫描差异** 三路信息融合，与当前 `PROJECT_CONTEXT.md` 进行对比，生成结构化差异报告，分三级：
   - 🔴 **重大差异**：技术栈变更、目录结构重大调整、核心约定推翻、新增/删除主要功能。
   - 🟡 **一般差异**：版本号微调、新增少量文件、待办项状态变更。
   - ⚪ **可自动同步**：严格仅为日期/时间戳、待办项勾选状态、需求完成度状态更新。

5. **自动更新上下文**（无需用户确认）：
   - 对于 **⚪ 可自动同步** 类差异，AI 直接修改 `PROJECT_CONTEXT.md`。
   - 对于 **🟡 一般差异**，AI 在更新文档的同时，需在推送前的最终报告里**简要列出所有变更项**，供用户快速审阅后推送。
   - 对于 **🔴 重大差异**（如技术栈变更、目录结构重大调整），AI 必须**列出变更项并等待用户确认**后再执行推送，避免文档与用户预期出现不可逆偏差。
   - 更新操作：
     - 更新 `last_updated` 和 `version`（补丁号递增）。
     - 修正各章节内容，保持原有格式。
     - 在“变更日志”顶部追加一条记录，摘要本次 push 涉及的新需求及修正。
     - 将本次对话中确定的**新增编码约定、已知问题、待办事项**等同步录入。

6. **执行 Git 提交流程**：
   - 先将更新后的 `PROJECT_CONTEXT.md` 单独 `git add`。
   - 建议提交信息格式：`docs(ctx): sync project context for push - [需求摘要]`
   - 可随后合并到用户原本要 push 的提交中，或作为独立提交随主提交一起推送（由 AI 根据用户习惯决策）。
   - 最后执行 `git push`。

### 2.3 AI 内部分析时的扫描清单（参考）
- 依赖与版本：对比实际锁文件/配置文件与文档第 2 章。
- 目录结构：检查实际顶级目录是否与文档第 3 章一致。
- 功能完成度：根据 Git 提交记录中的 `[完成]` 标记和对话确认，更新第 5 章。
- 待办项消化：若对话中解决了某个已记录的待办项，在第 7 章标记完成或删除。

### 2.4 异常与边缘情况
- 若项目根目录不存在 `PROJECT_CONTEXT.md`：AI 在 push 前**必须先引导用户创建**，基于扫描结果生成预填充模板，用户审阅保存后继续流程。
- 若文档损坏：AI 提示并由用户决定是否重建（可从 Git 历史恢复）。
- 若用户**仅要求 push 而没有任何代码或需求变更**：AI 仍执行扫描，但可判定“无需更新上下文”，直接推送。
- 若对话中用户明确要求“本次 push 不要更新上下文”：AI 须遵循，但应提示风险并记录在案。

## 3. 利用 Git 记录需求迭代（需求演进追踪）

### 3.1 目标与可行性
- **目标**：让 AI 通过读取 Git 日志，独立获取需求演进的全景图。
- **可行性**：`git log` 是零风险只读操作。标准化的提交信息可被 AI 精确解析，形成“需求时间线”。

### 3.2 Git 提交信息规范（必须遵守）
采用**约定式提交**格式，并附加需求追踪信息。

```
<type>(<scope>): <简短描述>

[需求ID][完成度] <详细说明>
```

- **type**：`feat`、`fix`、`refactor`、`docs`、`chore` 等。
- **scope**：可选，功能模块名。
- **正文建议内容**：
  - `[REQ-xx]`：需求编号，便于关联多次提交。
  - `[完成]` / `[部分]`：标记交付状态。
  - 来源或其他说明。

**示例**：
```
feat(auth): 增加手机验证码登录
[REQ-11][完成] 实现了短信验证码发送与验证接口，集成第三方短信服务。
```

> 此规范写入项目 `CONTRIBUTING.md` 或 `PROJECT_CONTEXT.md` 编码规范章节。

### 3.3 AI 如何使用 Git 记录
- 每次 push 前执行 `git log --oneline --grep="feat|fix|refactor" -20`，提取功能性提交。
- 按时间线识别需求实现情况，交叉验证文档第 5、7 章。
- 将未记录的需求补充进上下文，保证文档与代码仓库历史一致。

## 4. 自我迭代能力

### 4.1 文档的自我进化
在对话过程中，当发生以下情况时，AI 应**在内存中暂记**，待到 push 同步时统一提议/写入：
- 确定了一项新的编码约定。
- 决策引入新依赖或升级关键库。
- 识别到一个新的已知问题或技术债务。
- 用户口头描述了一个未记录的功能模块。

Push 时，AI 可输出：
> 📄 **上下文更新**：根据本次对话，已自动将“API 路由统一使用 `/api/v2/` 前缀”约定写入 `PROJECT_CONTEXT.md` 第 4 节。

### 4.2 版本与回溯
- 文档通过 Git 进行版本管理，每次 push 更新视为一次独立的文档版本演进。
- “变更日志”章节完整保留历史，不可删减。

## 5. 新窗口对话最佳实践

### 5.1 用户侧
- 开启新对话时，首条消息务必包含：
  ```
  请阅读项目根目录的 PROJECT_CONTEXT.md 作为上下文。
  文件路径：./PROJECT_CONTEXT.md
  ```
  或直接上传该文件。
- **推荐做法**：在 AI 编辑器（如 Cursor、Gemini Code Assist 等）中，将上述指令配置为**全局 System Prompt / User Rules**，从而在每次新会话时自动注入项目上下文路径，无需手动输入。

### 5.2 AI 侧
- 收到上下文后，AI 输出极简确认信息，如：“已读取上下文：MyProject v1.2.0，Next.js 14 + TypeScript。”
- 对话期间，AI **不主动检查实际代码差异**，完全信任当前提供的文档，除非用户要求或即将 push。
- 如果 AI 观察到用户指令与文档记录明显相悖（如提到一个文档中不存在的模块），可适时询问：“我注意到上下文未记录该模块，是否需要稍后推送时一并更新？”

---

## 6. 通用性保障与定制建议

本文档规范**适用于任何技术栈、任何规模的项目**。唯一需要按项目调整的是：
- `PROJECT_CONTEXT.md` 中“技术栈”“目录结构”等具体内容。
- push 前扫描项可根据项目实际配置文件（如 `Dockerfile`, `Makefile`, `.env.example`) 调整。
- Git 提交格式可微调，但需保持可解析性。

对于 **Monorepo**，根目录保留总览上下文，子包可维护 `PACKAGE_CONTEXT.md`，push 时分别同步。

## 7. 示例：一次典型的 Push 触发同步交互

```
用户：这段代码没问题了，帮我 push 到远程吧。
AI：收到 push 指令。开始执行推送前上下文同步…
    回顾本次对话，识别到新需求：
    - 新增用户头像上传功能
    - 引入 sharp@0.33.2 进行图片处理
    结合 git log 和实际扫描，确认已存在但未记录的：
    - feat(storage): 引入对象存储服务 (REQ-13, 已完成)
    与 PROJECT_CONTEXT.md (v1.2.0) 比对存在差异，已自动更新：
    🔴 目录新增 /services/storage，技术栈新增 sharp@0.33.2
    🟡 功能状态：头像上传标记为完成，对象存储服务标记为完成
    文档已更新至 v1.2.1，变更日志已追加。
    正在提交并推送…
    [exec] git add PROJECT_CONTEXT.md && git commit -m "docs(ctx): sync context - add avatar upload, storage service" && git push
    ✅ 推送成功，上下文已与代码同步。
```

> 此版本将**同步彻底收敛至 Push 时刻**，既保证了对话的连续性，又实现了代码、需求、文档三者的原子化对齐。可直接作为团队的正式规范启用。
