# AI Coding 工程约束

## 核心机制

以项目根目录下的 `PROJECT_CONTEXT.md` 为唯一事实来源，**在用户要求 git push 时自动完成差异分析、上下文更新与推送**，实现代码与文档的原子性同步。

## 新对话首条指令

开启新对话时，首条消息务必包含：
```
请阅读项目根目录的 PROJECT_CONTEXT.md 作为上下文。
文件路径：./PROJECT_CONTEXT.md
```

## Push 前自动同步流程

当用户指令包含 `git push` 时，按以下顺序执行：

1. **对话需求提取**：回顾本次对话，提取用户提出的所有实质性新增需求或改动意图
2. **Git 历史分析**：执行 `git log --oneline --grep="feat|fix|refactor" -15`，获取近期功能性提交
3. **实际项目扫描**：
   - 比对依赖文件版本与文档记录
   - 检查文档目录结构是否与现实一致
4. **综合差异生成**：分三级：
   - 🔴 重大差异：技术栈变更、目录结构重大调整
   - 🟡 一般差异：版本号微调、新增少量文件
   - ⚪ 可自动同步：日期/时间戳、待办项状态更新
5. **自动更新上下文**：⚪ 直接修改，🟡 列出变更项供审阅，🔴 等待用户确认
6. **执行 Git 提交流程**：`git add PROJECT_CONTEXT.md && git commit && git push`

## Git 提交规范

```
<type>(<scope>): <简短描述>

[需求ID][完成度] <详细说明>
```

- **type**：`feat`、`fix`、`refactor`、`docs`、`chore`
- **scope**：功能模块名
- **正文**：`[REQ-xx]` 需求编号，`[完成]`/`[部分]` 标记交付状态

## 编码规范

- 所有 SQL 使用参数化查询，禁止字符串拼接
- FastAPI 端点使用同步 `def`，BGE/Chroma/OpenAI 调用在线程池执行
- 异常处理分层：业务异常返回 4xx，系统异常返回 500 且不泄露敏感信息
- 数据库操作使用 `try/finally` 确保连接关闭
