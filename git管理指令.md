# Git 基础操作管理指南

## 📦 核心 Workflow 类比：寄快递

想象你要寄一堆文件给 GitHub：

| Git 命令 | 寄快递类比 |
|:---|:---|
| `git add .` | 把文件**打包、装箱**，准备寄出（还没寄） |
| `git commit -m "说明"` | 在包裹上**贴标签、写备注**，然后**封箱存档**到本地仓库 |
| `git push` | 把封好的包裹**真正寄出去**（上传到 GitHub） |

---

## 🗂️ Git 的三个区域

Git 管理代码时，文件会在三个区域之间流转：

```
┌─────────────────┐     git add      ┌─────────────────┐    git commit    ┌─────────────────┐
│   工作区         │ ───────────────▶ │    暂存区        │ ──────────────▶ │   本地仓库       │
│ (你编辑的文件)    │                  │ (准备提交的内容)  │                 │ (版本历史存档)   │
└─────────────────┘                  └─────────────────┘                 └─────────────────┘
                                                                              │
                                                                              │ git push
                                                                              ▼
                                                                        ┌─────────────────┐
                                                                        │   远程仓库       │
                                                                        │ (GitHub 服务器)  │
                                                                        └─────────────────┘
```

---

## 🔧 核心命令详解

### 1. git add .

**作用**：把当前工作区的**所有修改**（新增、删除、修改的文件）放入暂存区。

- `.` 表示"当前目录下的所有文件"
- 如果只想添加某个文件：`git add README.md`
- 这是**可撤销**的，还没真正保存

**什么时候用**：你改完代码，准备打包这次修改时。

---

### 2. git commit -m "你的修改说明"

**作用**：把暂存区的内容**正式保存**到本地 Git 仓库，生成一个永久的历史记录（快照）。

- `-m` 是 message 的缩写，后面写这次修改的描述
- 描述要清晰，例如：
  - `"修复用户登录闪退问题"`
  - `"新增知识库文档上传功能"`
  - `"优化 RAG 检索速度"`
- 执行后，本地就多了一个"版本"，随时可以回退到这个状态

**什么时候用**：确认某批修改已完成、有意义，要存档时。

---

### 3. git push

**作用**：把本地仓库里**新的提交**上传到远程仓库（GitHub）。

- 只有执行了 `push`，GitHub 上才能看到你最新的代码
- 第一次推送需要 `git push -u origin main`，之后直接 `git push` 即可

**什么时候用**：本地 commit 完成后，想同步到 GitHub 时。

---

## 🔄 完整日常工作流示例

假设你修改了 `README.md`，新增了一个 `app.py`：

```bash
# 1. 查看当前状态（哪些文件被改了）
git status

# 2. 把修改打包到暂存区
git add README.md app.py

# 3. 确认保存到本地仓库，写清楚这次做了什么
git commit -m "新增 app.py 主程序，更新项目说明"

# 4. 上传到 GitHub
git push
```

## 📋 其他常用命令速查表

| 命令 | 作用 |
|:---|:---|
| `git status` | 查看当前状态（哪些文件改了、哪些在暂存区） |
| `git log` | 查看提交历史（按 `q` 退出） |
| `git diff` | 查看具体修改了哪些内容 |
| `git pull` | 从 GitHub **拉取最新代码**到本地（团队协作前必做） |
| `git clone [网址]` | 把别人的仓库下载到本地 |
| `git rm --cached [文件]` | 把文件从暂存区撤回（取消 add） |
| `git remote -v` | 查看关联的远程仓库地址 |
| `git branch` | 查看当前分支 |

---

## ⚠️ 重要习惯

1. **先 pull 再 push**：多人协作时，推送前先 `git pull` 拉取最新代码，避免冲突
2. **commit 粒度要小**：一个功能一个 commit，不要攒几十个文件的修改一次性 commit
3. **commit 信息要写清楚**：方便以后查看历史时知道每次改了什么
4. **不要强推公共分支**：`git push -f` 只在个人仓库的初始阶段使用，团队协作时强制推送会覆盖别人的代码

---

## 🎯 记住口诀

> **add → commit → push**
>
> 修改打包 → 本地存档 → 上传云端
