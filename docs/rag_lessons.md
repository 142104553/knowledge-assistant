# RAG 实战经验：为什么知识库越大，召回越差

> 来源：实际项目踩坑记录

## 现象

把调度规程、继电保护手册、配网运维规范、设备说明书全扔一个知识库里。

问继电保护的问题，检索结果里混进来三条配网的、两条调度的。关键词都命中了，但答非所问。

## 错误路径：硬调 TopK

| TopK | 效果 | 问题 |
|:---:|:---|:---|
| 10 | 真正需要的漏了 | 召回不足 |
| 20 | 勉强能用 | 噪音开始混入 |
| 30 | 命中率一般 | LLM 上下文被稀释 |
| 40 | 勉强能捞到 | 延迟高、token 贵、LLM 被带偏 |

**结论**：TopK 硬调是治标不治本。语义空间被多个领域挤占，向量相近不代表语义相关。

## 正确路径：语义空间隔离

### 核心洞察

> 一锅炖不如分灶做。

每个领域的文档有自己的术语体系和语义空间。混在一起时，BGE 会把"保护定值"和"保护接地"当成相似的，因为都有"保护"两个字。

分开存储后：
- 保护知识库里只有保护相关的 chunk
- 配网知识库里只有配网相关的 chunk
- 互不干扰

### 改造后效果

| 指标 | 一锅炖 TopK=40 | 分库 TopK=10 |
|:---:|:---:|:---:|
| 召回准确率 | 低 | **高** |
| 平均延迟 | 高（40条上下文） | **低**（10条上下文） |
| Token 消耗 | 高 | **低** |
| LLM 幻觉率 | 高（被无关内容带偏） | **低** |

### 实现方式

**轻量版：元数据过滤**
```python
# 上传时打标签
chunk.metadata["domain"] = "继电保护"

# 查询时过滤
vector_store.similarity_search(
    filter_dict={"domain": "继电保护"},
    top_k=10
)
```

**彻底版：分 Collection + 领域路由**
```
用户提问 → 领域分类器 → 对应 Agent → 对应 Collection → 回答
```

## 关键认知

1. **向量相似度 ≠ 语义相关**： Embedding 模型会把表面相似的词（如"保护"）映射到相近向量，但领域不同。
2. **LLM 只能读你给它的**：上下文里有噪音，输出就会被污染。过滤要在进 LLM 之前做。
3. **组织方式比调参重要**：折腾权重、阈值、重排序，不如把知识库拆清楚。
